# 報告撰寫（六階段流程）

## 階段零：逐一釐清（最重要）

不要一次問一堆問題。**每次只問一個問題，等使用者回答後再問下一個。**

依序釐清以下資訊（如果 $ARGUMENTS 已經包含的就跳過）：

**第 1 問：讀者**
「這份報告是給誰看的？他的角色和技術水平是？」

等回答。

**第 2 問：目的**
「這份報告要幫讀者做什麼決定？或回答什麼問題？」

等回答。

**第 3 問：讀者的顧慮**
「你覺得讀者看完最可能問什麼？或他最擔心什麼？」

等回答。

**第 4 問：資料**
「我需要參考哪些資料？本地檔案、網路搜尋、還是你會提供？」

等回答。

**第 5 問：格式**
「字數大約多少？語言？有沒有特別的格式要求？」

等回答。

**第 6 問：遺漏檢查**
根據前 5 個回答，判斷有沒有關鍵遺漏。如果有，再問一個針對性問題。例如：
- 讀者說是主管，但沒提預算 →「主管對預算有沒有上限或敏感度？」
- 需求審查但沒提時程 →「這個需求有沒有交付期限？」
- 提案但沒提替代方案 →「主管知道有其他選項嗎？還是只看你這個方案？」

如果沒有遺漏，直接告訴使用者：「資訊已足夠，我開始研究和寫大綱。」

**規則：**
- 每次只問一個問題
- 問題要具體，不要問「還有什麼要補充的嗎」
- 使用者的回答如果模糊，追問一次就好，不要反覆追問
- 5-6 個問題內結束，不要問超過 7 個

## 階段一：定義

將階段零收集到的資訊整理成報告定義：

```
讀者：___
核心問題：___
讀者會問的問題：
1. ___
2. ___
3. ___
格式：___
資料來源：___
```

不需要再問使用者確認，直接進入下一階段。

## 階段二：研究

根據資料來源：
- 本地檔案 → 用 Read / Grep / Glob
- 網路 → 用 WebFetch
- 多來源 → 用 Agent 平行研究

將研究結果寫入 `/tmp/report-research.md`，每個來源附 URL。

## 階段三：大綱

根據研究結果建立大綱，每章標註：
- 一句話說明
- 預估字數
- 引用的資料來源
- 回答讀者的哪個問題

**輸出大綱後暫停，等使用者確認再繼續。**

## 階段四：撰寫

分段寫入 `/tmp/report-draft.md`：
- 每次寫 1-2 個章節
- 每章控制在 500-800 字
- 數據用 bash 計算，不靠記憶
- 引用標註來源

## 階段五：審查

讀取 /tmp/report-draft.md，逐項檢查：
1. 讀者會問的每個問題都有被回答嗎？
2. 每章是否回答了大綱問題？
3. 有無沒有根據的斷言？
4. 摘要能否 30 秒看懂結論？
5. 有沒有讀者不懂的術語？
6. 字數是否達標？

列出問題 → 修改 → 搬到正式位置 → git commit + push。
