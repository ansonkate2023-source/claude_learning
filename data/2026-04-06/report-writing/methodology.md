# 用 Claude Code 寫報告的方法論

## 核心理念

報告寫作不是一步完成的。拆成 5 個階段，每個階段有明確的輸入和輸出。
Claude Code 的優勢在於：**每個階段都能用不同工具組合**，而不是只靠對話。

---

## 階段一：定義（5 分鐘）

在動手之前，先回答四個問題：

```
1. 報告給誰看？（讀者是誰、技術水平、關心什麼）
2. 報告要回答什麼問題？（不超過 3 個核心問題）
3. 報告的格式要求？（長度、語言、結構、是否需要圖表）
4. 資料來源有哪些？（本地檔案、網路、API、資料庫）
```

然後**最關鍵的一步**——列出讀者會問的問題：

```
5. 這個讀者看完會問什麼？（列 5 個，報告必須全部回答）
```

### 不同讀者的典型問題

#### 技術主管

| 他們問的 | 你的報告要回答 |
|----------|---------------|
| 結論是什麼？要我做什麼決定？ | 摘要第一段就給結論和建議行動 |
| 風險在哪？最壞情況是什麼？ | 專門一節講風險和已知限制 |
| 要多少資源？多長時間？ | 給具體數字，不要「大約」「可能」 |
| 跟現有方案比，好在哪？ | 用表格比較，不要只講優點 |
| 如果不做會怎樣？ | 說明不行動的成本 |

#### 不懂技術的客戶

| 他們問的 | 你的報告要回答 |
|----------|---------------|
| 這對我的業務有什麼影響？ | 用業務語言，不用技術術語 |
| 要花多少錢？ | 明確的費用或費用範圍 |
| 什麼時候能看到結果？ | 具體的時間表或里程碑 |
| 安全嗎？可靠嗎？ | 用類比解釋，「就像銀行的保險箱」 |
| 你能給我看個例子嗎？ | 附上截圖、demo、或前後對比 |
| 我需要做什麼？ | 明確列出他們的 action items |

#### 團隊成員

| 他們問的 | 你的報告要回答 |
|----------|---------------|
| 這跟我的工作有什麼關係？ | 說明對每個角色的影響 |
| 我要改變什麼？ | 具體的流程變更清單 |
| 有沒有範例可以參考？ | 附上可直接使用的模板或程式碼 |
| 遇到問題找誰？ | 列出負責人和溝通管道 |

### 實際操作

告訴 Claude：

```
我要寫一份報告給 {讀者}。
他們最可能問的 5 個問題是：
1. ...
2. ...
3. ...
4. ...
5. ...

報告的每個章節必須回答至少一個問題。
如果某個問題沒有被回答到，提醒我。
```

**為什麼這步最重要：** 報告的價值不在於你寫了什麼，在於讀者的問題有沒有被回答。先列問題再寫內容，保證不會「寫了很多但讀者最想知道的沒講」。

### 範例

```
讀者：技術主管，熟悉 AI 但不寫程式
核心問題：Claude Code 的 skill 測試框架是否成熟可用？
格式：正體中文，2000-3000 字，含表格和建議
資料來源：本專案的 design-process.md、SKILL.md、測試報告

讀者會問的 5 個問題：
1. 這個測試框架現在能用嗎？成熟度如何？
2. 用了之後能節省多少時間？
3. 有什麼限制和風險？
4. 需要投入什麼資源來導入？
5. 不導入的話，現在怎麼測 skill？
```

## 階段二：研究（10-30 分鐘）

根據資料來源類型，選擇工具組合：

### 本地檔案研究

```
# 讀取特定檔案
直接請 Claude 讀取：「讀取 data/2026-04-06/skill-development/ 下所有 .md 檔案」

# 搜尋關鍵字
用 Grep 搜尋：「在整個專案中搜尋 '測試框架' 相關內容」

# 掃描專案結構
用 Glob 找檔案：「列出所有 .yaml 測試檔案」
```

### 網路研究

```
# 單一主題深入
「用 WebFetch 讀取 https://example.com/article 並摘要重點」

# 多來源平行研究（用 Agent）
「同時從 3 個來源研究 Claude Code skill 的最佳實踐：
 1. Anthropic 官方文件
 2. Hacker News 最新討論
 3. Google News 相關報導」
```

### 研究輸出

將研究結果寫入暫存檔，不要只記在對話中：

```
把研究結果寫入 /tmp/report-research.md，包含：
- 每個來源的重點摘要（3-5 句）
- 直接引用的段落
- 來源 URL
```

**為什麼寫檔案：** 對話 context 會被壓縮，但檔案不會。後續寫作時可以隨時回頭讀。

## 階段三：大綱（5 分鐘）

根據研究結果，建立報告骨架：

```
根據 /tmp/report-research.md 的內容，為以下報告建立大綱：
- 讀者：{讀者}
- 核心問題：{問題}
- 格式：{格式}

要求：
1. 每個章節一句話說明要寫什麼
2. 標註每個章節的預估字數
3. 標註每個章節需要引用的資料來源
```

### 大綱範例

```markdown
# 報告大綱：Claude Code Skill 測試框架評估

## 1. 摘要（200 字）
一段話總結結論：框架可用但有明確限制。

## 2. 背景（300 字）
為什麼需要測試 skill？問題是什麼？
來源：design-process.md 步驟 1-2

## 3. 框架設計（500 字）
四種測試類型、bash grep 驗證、結果持久化。
來源：SKILL.md、test-greet.yaml

## 4. 實測結果（400 字）
第一次實際執行的結果和發現。
來源：results/2026-04-06-2230.md

## 5. 迭代教訓（500 字）
v1-v8 的演進過程和每次發現的問題。
來源：design-process.md 步驟 6-9

## 6. 已知限制（300 字）
Skill tool 不是函數呼叫、非確定性輸出等。
來源：SKILL.md 限制章節

## 7. 建議（300 字）
下一步該做什麼。
```

**確認大綱後再繼續。** 這是最重要的檢查點——結構錯了，內容再好也沒用。

## 階段四：撰寫（20-60 分鐘）

### 策略：分段寫，不要一次寫完

```
根據大綱，先寫第 1-2 章。
參考 /tmp/report-research.md 中的資料。
寫入 /tmp/report-draft.md。
```

寫完前兩章後：

```
繼續寫第 3-4 章，接在 /tmp/report-draft.md 後面。
```

**為什麼分段：**
1. Claude 的單次輸出有長度限制，一次寫 3000 字可能被截斷
2. 分段可以在每段之間檢查品質
3. 避免後半段因為 context 膨脹而品質下降

### 撰寫指令的寫法

**差的指令：**
```
寫一份關於 skill 測試的報告
```

**好的指令：**
```
撰寫報告的第 3 章「框架設計」：
- 讀者是技術主管，不需要解釋什麼是 grep
- 重點解釋為什麼選擇 bash grep 而非 Claude 判斷
- 引用 design-process.md 中「確定性工具優先」的原則
- 用表格比較四種測試類型
- 控制在 500 字以內
- 正體中文
```

### 長報告的 context 管理

報告超過 2000 字時，context 會開始膨脹。策略：

1. **研究結果在檔案中，不在對話中** — 已在階段二做到
2. **每寫完一大段，用 `/compact` 壓縮對話**
3. **需要回顧時讀檔案，不要靠記憶** — 「讀取 /tmp/report-draft.md 的第 3 章」

## 階段五：審查與定稿（10 分鐘）

### 自動審查清單

```
讀取 /tmp/report-draft.md，用以下清單審查：

1. **讀者問題覆蓋：** 階段一列的 5 個問題，每個都有被回答嗎？哪個沒有？
2. 每個章節是否回答了大綱中定義的問題？
3. 有沒有沒有根據的斷言？（每個重要結論需要來源）
4. 讀者能在 30 秒內從摘要了解結論嗎？
5. 有沒有使用讀者不懂的術語？（非技術讀者特別注意）
6. 有沒有多餘的章節可以刪掉？
7. 字數是否在要求範圍內？

列出需要修改的地方，不要直接改。
```

### 修改

根據審查結果，逐項修改：

```
修改 /tmp/report-draft.md：
1. 第 2 章的第二段缺少來源引用，加上 design-process.md 的引用
2. 摘要太長，從 300 字縮到 200 字
3. 第 5 章重複了第 3 章的內容，刪除重複部分
```

### 搬到正式位置

```
把 /tmp/report-draft.md 的最終版本寫入
data/2026-04-06/reports/skill-testing-evaluation.md
```

### git 提交

```
git add data/2026-04-06/reports/
git commit -m "新增：Skill 測試框架評估報告"
git push
```

## 進階技巧

### 1. 用 Skill 封裝常用報告格式

如果你經常寫同類型的報告（週報、技術評估、競品分析），封裝成 skill：

```yaml
# .claude/skills/weekly-report/SKILL.md
---
name: weekly-report
description: 產生每週技術學習週報
disable-model-invocation: true
argument-hint: "[主題]"
---

產生本週的技術學習週報：

1. 讀取 data/ 下本週的所有 summary.md
2. 整理為以下格式：
   - 本週重點（3-5 條）
   - 新學到的技巧
   - 待深入的主題
   - 下週計畫
3. 寫入 reports/weekly-{date}.md
4. git commit + push
```

使用：`/weekly-report Claude Code`

### 2. 用 Agent 平行研究加速

```
同時啟動 3 個 Agent 研究：
1. Agent 1：搜尋 Anthropic 官方文件中關於 {主題} 的內容
2. Agent 2：搜尋 Hacker News 最近關於 {主題} 的討論
3. Agent 3：讀取本專案中相關的檔案

每個 Agent 把結果寫入 /tmp/research-{N}.md
```

### 3. 用 bash 做資料分析再寫報告

```bash
# 統計本月收集了多少篇文章
find data/2026-04-06 -name "raw.json" -exec cat {} + | grep -c '"title"'

# 統計各分類的文章數
for dir in data/2026-04-06/*/; do
  count=$(cat "$dir/raw.json" 2>/dev/null | grep -c '"title"' || echo 0)
  echo "$(basename $dir): $count"
done
```

把 bash 輸出的數據直接引用到報告中，比 Claude 自己數更準確。

### 4. 迭代式撰寫（適合複雜報告）

```
第一輪：寫出所有章節的草稿（不追求完美）
         ↓
第二輪：審查邏輯流暢度，調整章節順序
         ↓
第三輪：潤飾文字，確認引用和數據
         ↓
定稿
```

每輪之間用 `/compact` 壓縮，避免 context 爆炸。

## 常見錯誤

| 錯誤 | 改正 |
|------|------|
| 一句話就要 Claude 寫完整份報告 | 分階段、分章節、給明確指令 |
| 研究資料只存在對話中 | 寫入暫存檔案，對話會被壓縮但檔案不會 |
| 寫完不審查就提交 | 用審查清單逐項檢查 |
| 報告太長一次寫完 | 分段寫，每段控制在 500-800 字 |
| 數據靠 Claude 記憶 | 用 bash 算，結果更準確 |
| 每次寫報告都從零開始 | 封裝成 skill，建立可重複的模板 |

## 快速參考卡

```
寫報告的 5 步：

1. 定義 → 讀者是誰？回答什麼問題？
2. 研究 → WebFetch / Read / Agent → 寫入 /tmp/research.md
3. 大綱 → 章節 + 字數 + 來源 → 確認後再繼續
4. 撰寫 → 分段寫入 /tmp/draft.md → 每段 /compact
5. 審查 → 清單檢查 → 修改 → 搬到正式位置 → git push
```
