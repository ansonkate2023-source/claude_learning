# Skill 測試框架 — 從約束出發的設計

## 步驟 1：不可行清單

在 Claude Code 中測試 skill，以下事情**做不到或不可靠**：

| 約束 | 原因 | 影響 |
|------|------|------|
| 無法確定性地捕獲 skill 輸出 | 輸出混雜推理、格式化、對話語氣 | 字串精確比對不可靠 |
| 無法查詢「哪個 skill 被載入」 | 沒有 API | 無法直接測試自動觸發 |
| 無法在同一 session 隔離測試 | 對話歷史累積 | Claude 知道自己在被測試 |
| 子代理隔離對話但共享檔案系統 | 架構限制 | 副作用測試仍有交叉風險 |
| YAML 由 Claude 解讀非程式解析 | 沒有 yaml parser | 格式錯誤可能靜默失敗 |
| LLM 輸出非確定性 | 同一 prompt 不同結果 | 測試可能時過時不過 |
| 同模型評審有盲點 | Claude 判 Claude | 品質退步可能漏掉 |
| 每個子代理消耗大量 token | 架構成本 | 測試數量受限 |

## 步驟 2：重新定義問題

「測試一個 skill」到底要回答什麼問題？

### 必須回答的問題（按價值排序）

1. **這個 skill 能用嗎？** — 呼叫後有合理回應，不會崩潰
2. **它做了該做的事嗎？** — 產出的檔案、執行的動作是正確的
3. **它沒做不該做的事嗎？** — 沒有洩露 prompt、沒有修改不該動的檔案
4. **改了之後有沒有變差？** — 回歸保護

### 不需要（或無法）回答的問題

- 輸出的文字是否「好看」→ 太主觀，不適合自動化
- skill 是否在正確時機被自動觸發 → 做不到直接驗證
- 回應速度夠不夠快 → 無法可靠計時

## 步驟 3：最小可行設計

只針對能可靠回答的問題設計，不多做。

### 測試類型 A：呼叫測試（回答問題 1）

**做法：** 在子代理中呼叫 skill，檢查是否正常完成。
**可靠度：** 高。子代理完成 = 沒崩潰。
**成本：** 每次一個子代理。

### 測試類型 B：副作用測試（回答問題 2）

**做法：** skill 執行後，用 bash 指令驗證檔案系統狀態。
**可靠度：** 最高。bash 退出碼是確定性的。
**成本：** 極低（bash 指令）。
**限制：** 只適用於有副作用的 skill（產出檔案、修改設定等）。
純文字回應的 skill 無法用此方法。

### 測試類型 C：邊界測試（回答問題 3）

**做法：** 用已知的攻擊 payload 呼叫，檢查輸出不包含敏感內容。
**可靠度：** 中。字串排除檢查是確定性的，但攻擊者可能繞過。
**成本：** 每次一個子代理。

### 測試類型 D：差異檢測（回答問題 4）

**做法：** 保存一份「已知好的輸出」，每次跑完用可量化的規則比較。
**可靠度：** 中。結構化比較比語意判斷好，但仍有盲點。
**成本：** 低（比較邏輯簡單）。
**規則（不依賴 LLM 判斷）：**
- 長度比：新 / 舊 在 0.5 ~ 2.0 之間
- 關鍵詞覆蓋：舊輸出中的名詞，新輸出覆蓋 ≥ 70%
- 結構保持：標題數量差 ≤ 2

### 不做的事

- ❌ 不做「輸出品質評分」— 不可靠
- ❌ 不做「工具追蹤」— 需要額外設定且 log 無法區分測試
- ❌ 不做「交叉模型評審」— 設定太複雜，收益不明確
- ❌ 不做「自然語言觸發測試」— 無法直接驗證，間接驗證也不可靠

## 步驟 4：心智模擬

模擬 Claude 讀到 SKILL.md 後的實際執行過程：

### 場景：/test-skill greet

1. Claude 讀 SKILL.md → 知道要讀 .claude/skill-tests/test-greet.yaml
2. Claude 用 Read 讀 YAML → 理解有 3 個 smoke + 5 個完整測試
3. 對 smoke test 1：
   - Claude 啟動子代理，prompt = "執行 /greet Alice 並回報結果"
   - 子代理執行 /greet Alice → 產出回應
   - 子代理回傳結構化結果
   - **問題：子代理能不能呼叫 Skill tool？** → 能，因為 allowed-tools 設定
   - **問題：子代理回傳的格式是否穩定？** → 不保證，需要容錯
4. Claude 收到結果 → 逐項檢查 expect/contract
   - **問題：Claude 是在用字串函數比較還是用「我覺得」？** → 沒有字串函數，只能靠 Claude 自己判斷 "contains"
   - **這意味著 contains 檢查本身就不是 100% 確定的**
5. 全部跑完 → 寫報告

### 心智模擬發現的新問題

- **contains 檢查是 Claude 做的，不是程式做的** → 有可能誤判
- **子代理回傳格式不保證是 JSON** → 需要容錯處理
- **8 個測試 = 8 個子代理 = 大量 token 消耗** → 需要控制數量

## 步驟 5：接受限制後的最終設計

### 核心認知

這個框架是一個**輔助工具**，不是一個測試引擎。
它能做到的是：讓你在改 skill 後快速跑一遍確認沒有明顯問題。
它做不到的是：保證 skill 在所有情況下都正確。

### 四種測試，各有明確的可靠度

| 類型 | 做法 | 可靠度 | 適用場景 |
|------|------|--------|----------|
| smoke | 子代理呼叫，完成就通過 | ★★★★ | 所有 skill |
| side-effect | bash 驗證檔案/狀態 | ★★★★★ | 有副作用的 skill |
| boundary | 攻擊 payload + 排除檢查 | ★★★ | 安全敏感的 skill |
| diff | 結構化數值比較 | ★★★ | 頻繁修改的 skill |

### 將 contains 檢查交給 bash 而非 Claude

把字串驗證從 Claude 的判斷改為 bash 的 grep：

```
# 與其讓 Claude 判斷「回應是否包含 Alice」
# 不如讓子代理把回應寫入檔案，再用 grep 檢查

子代理 → 寫回應到 /tmp/test-output-{id}.txt
主代理 → grep -q "Alice" /tmp/test-output-{id}.txt && echo PASS || echo FAIL
```

這讓字串檢查變成 100% 確定性的。

## 步驟 6：v4 → v5 Review（套用約束分析流程）

### 心智模擬發現的 v4 問題

| # | 問題 | 嚴重度 | 修正 |
|---|------|--------|------|
| A | SKILL.md 246 行太長，Claude 後半段執行偏離 | 高 | 縮短到 ~120 行，砍掉冗餘說明 |
| B | 子代理 prompt 要「執行 skill + 寫檔」兩件事，容易漏 | 高 | prompt 只要求兩步：執行、寫檔，措辭更直接 |
| C | grep 沒用 -F，`$ARGUMENTS` 中的 `$` 被當 regex | 高 | 全部改用 `grep -F`（fixed string） |
| D | `excludes: ["---\nname:"]` 跨行匹配永遠通過 | 高 | 拆成多個單行條件 |
| E | 快照「中文關鍵詞提取」bash 做不到 | 高 | 砍掉關鍵詞覆蓋，只比長度和行數 |
| F | 每個 smoke 一個子代理太貴 | 中 | 保留但建議控制數量 |

### v5 設計原則

1. **SKILL.md 精簡** — 只寫 Claude 執行時需要的指令，不寫教學說明
2. **grep -F 不用 regex** — 所有字串搜尋用 fixed string 避免特殊字元問題
3. **不做做不到的事** — 砍掉中文關鍵詞提取，快照只比長度/行數
4. **跨行條件拆開** — 不依賴 grep 的多行匹配能力
5. **子代理 prompt 單純** — 「執行 skill，寫檔案」兩步，不加額外要求
