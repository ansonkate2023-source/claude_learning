# Skill 開發方法論 — 從 8 次迭代中提煉

本文件從 test-skill 框架的 8 個版本（v1→v8）+ 1 次實際執行中，提煉出可重複使用的 skill 開發方法論。

---

## 一、開發流程

### 1. 先寫一個最小 skill，立刻跑

```yaml
---
name: my-skill
description: 一句話描述做什麼+何時觸發
---

最簡單的指令，3-5 行。
```

跑一次 `/my-skill test-input`，看 Claude 實際怎麼回應。
**不要在第一版就加進階功能。**

### 2. 從真實使用中發現問題

用 skill 處理 2-3 個真實案例，記錄：
- 哪裡符合預期？
- 哪裡偏離？
- 什麼情況觸發了，什麼情況沒觸發？

### 3. 針對問題修改，不堆功能

每次只修一個問題。改完再跑一次確認。

### 4. 穩定後寫測試

用 test-greet.yaml 格式寫 smoke + 3-5 個關鍵測試。
跑一次 `/test-skill my-skill` 建立基線。

## 二、設計原則（從失敗中學到的）

### 原則 1：能交給確定性工具的事，不要靠 Claude

| 操作 | ❌ 靠 Claude | ✅ 靠工具 |
|------|-------------|----------|
| 字串包含檢查 | Claude 判斷「有沒有 Alice」 | `grep -qiF 'Alice' file.txt` |
| 記住多個結果 | Claude context 記憶 | `echo "PASS" >> results.txt` |
| 建立唯一路徑 | Claude 自己想一個 | `mktemp -d /tmp/xxx-XXXXXX` |
| 比較檔案長度 | Claude 目測 | `wc -m < file.txt` |

### 原則 2：列假設再驗證，不要「想一下覺得對」

每次設計新功能前：
1. 寫下「我相信 X 為真」
2. 能用 bash 實測的就跑一次
3. 不能實測的標記為「未驗證」

導致 v1-v6 失敗的全是**未列出的假設**：
- `grep` 會正確處理 `$` → 錯（需要 `-F`）
- 子代理會載入 skill → 錯（它會繞過）
- `date +%s` 兩次呼叫同值 → 錯（可能不同秒）

### 原則 3：心智模擬要走三層

| 層次 | 問什麼 | 範例 |
|------|--------|------|
| 架構 | 整體流程邏輯對嗎？ | 「子代理能載入 skill 嗎？」→ 不能 |
| 執行 | 每一步能跑嗎？ | 「shell 變數跨 Bash tool 持久嗎？」→ 不持久 |
| 語法 | bash 指令對嗎？ | 「grep -i 對 excludes 合適嗎？」→ 不合適 |

只做一層的模擬，必定會遺漏其他層的問題。

### 原則 4：實際跑一次 > 理論 review 十次

v1-v8 都是理論推演。實際跑一次才發現最重要的事實：**Skill tool 每次呼叫需要使用者互動**。

流程：設計 → 建造 → **馬上跑一次** → 從真實結果改進。

### 原則 5：兩個好目標可能互相矛盾

v3-v5 同時追求「測試隔離」和「確保 skill 被載入」，結果兩個都做不好。

遇到矛盾時：明確取捨，不要假裝兼顧。

### 原則 6：接受限制，標注清楚

不可修正的限制就寫在文件裡，不假裝完美：
- Skill tool 不是函數呼叫，寫入檔案的文字是手動複製
- 快照只比長度和行數，不偵測微妙退步
- 非確定性輸出，同一測試可能時過時不過

## 三、測試策略

### 測試金字塔

```
        /  對抗  \        ← 少量：injection、XSS
       / 結構契約 \       ← 中量：長度、行數、標題
      / bash 副作用 \     ← 若有：檔案存在、退出碼
     / 字串 contains \    ← 核心：grep -F 驗證關鍵內容
    / ── smoke ── \       ← 底層：跑過不崩就好
```

從底層開始寫，往上加。大部分 skill 只需要 smoke + contains 就夠了。

### 測試數量建議

| skill 複雜度 | smoke | 完整測試 | 對抗 | 總計 |
|-------------|-------|---------|------|------|
| 簡單（greet） | 3 | 2-3 | auto(3) | ~9 |
| 中等（deploy） | 3 | 5-7 | auto(3) + 2 | ~15 |
| 複雜（batch） | 不建議測 | — | — | — |

複雜 skill（涉及 git worktree、多 agent）不適合用這個框架測。手動驗證更可靠。

### contains vs excludes 的大小寫選擇

```
contains 用 grep -iF（不分大小寫）→ 因為找的是「有沒有這個概念」
excludes 用 grep -F （分大小寫）→ 因為找的是「有沒有洩露精確的字串」
```

## 四、SKILL.md 撰寫技巧

### description 的黃金法則

前 250 字元最重要（會被截斷）。格式：`做什麼 + 何時觸發`。

```yaml
# ❌
description: 處理文件

# ✅
description: 從 PDF 提取文字和表格。使用於處理 PDF 或提到表單時。
```

### 指令長度控制

| 長度 | 效果 |
|------|------|
| < 50 行 | 最佳。Claude 能完整執行 |
| 50-150 行 | 可接受。後半段可能偏離細節 |
| > 150 行 | 危險。拆到子檔案 |

### 自由度匹配任務風險

| 任務 | 自由度 | 寫法 |
|------|--------|------|
| 程式碼審查 | 高 | 「分析程式碼，提出建議」 |
| 格式轉換 | 中 | 提供模板 + 「可依情況調整」 |
| 資料庫遷移 | 低 | 「執行這個指令，不要修改」 |

## 五、完整開發 checklist

### 建立新 skill
- [ ] 一句話 description（做什麼 + 何時觸發）
- [ ] 最簡指令（< 50 行）
- [ ] 立刻跑一次
- [ ] 用 2-3 個真實案例測試
- [ ] 根據結果修改

### 穩定後加測試
- [ ] smoke 3 個（正常、空輸入、特殊字元）
- [ ] contains/excludes 2-3 個
- [ ] 跑一次 `/test-skill` 建立基線
- [ ] 建立快照

### 修改 skill 後
- [ ] 跑 `/test-skill` 確認沒有回歸
- [ ] 檢查快照是否大幅變化
