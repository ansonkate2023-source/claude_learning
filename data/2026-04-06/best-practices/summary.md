# 官方最佳實踐 - 2026-04-06

來源：Anthropic 官方文件 platform.claude.com

## 一、輸出控制

### 溝通風格
Claude 4.6 的預設風格更簡潔自然：
- **更直接務實** — 基於事實的進度報告，不自我吹噓
- **更口語化** — 減少機器感
- **更精簡** — 可能跳過冗長摘要

如果需要更多可見性：
```
完成涉及工具使用的任務後，提供一個簡短的工作摘要。
```

### 格式控制四法
1. **說「要做什麼」而非「不要做什麼」**
2. **用 XML 標籤指定格式區域**
3. **讓 prompt 風格匹配期望的輸出風格**
4. **明確的格式指引**（需要時）

## 二、工具使用

### 明確要求行動
```
❌ 你能建議一些改善嗎？（Claude 只會建議）
✅ 修改這個函數以提升效能。（Claude 會動手改）
```

### 主動行動 vs 保守行動
**想要 Claude 主動：**
```
預設執行更改而不只是建議。如果使用者意圖不明確，
推斷最有用的行動並執行。
```

**想要 Claude 保守：**
```
除非明確指示，不要直接修改檔案。
預設提供資訊和建議，而非採取行動。
```

## 三、思考與推理

### Adaptive Thinking（自適應思考）
Claude 4.6 會根據問題複雜度自動調整思考深度：
- **高 effort** — 深度推理，適合複雜任務
- **中 effort** — 平衡品質與速度
- **低 effort** — 快速回應，適合簡單查詢

### 避免過度思考
```
選定一個方法後就執行。避免反覆重新評估決策，
除非遇到直接矛盾的新資訊。
```

### 自我檢查
```
在完成之前，根據 [測試標準] 驗證你的答案。
```

## 四、Agent 系統

### 平衡自主與安全
```
考慮行動的可逆性和影響範圍。
本地可逆操作（編輯檔案、執行測試）可自由執行，
但難以逆轉或影響共享系統的操作，請先確認。

需確認的操作範例：
- 刪除檔案或分支
- git push --force
- 在 PR 或 issue 上留言
- 修改共享基礎設施
```

### 研究與資訊收集
```
以結構化方式搜尋資訊。
收集資料時建立多個競爭假設。
在進度筆記中追蹤信心水平。
定期自我批評你的方法和計畫。
```

## 五、防止常見問題

### 防止過度工程
```
避免過度工程。只做被要求或明確必要的變更。
- 不要添加未被要求的功能
- 不要為未修改的程式碼添加文件
- 不要為不可能的情境添加錯誤處理
- 不要為一次性操作建立抽象
```

### 防止硬編碼
```
寫一個通用解決方案。不要硬編碼值或建立只適用於特定測試的方案。
實作正確解決問題的實際邏輯。
```

### 減少幻覺
```
絕不對未打開的程式碼做推測。
如果使用者提到特定檔案，必須先讀取再回答。
確保回答有根據，避免幻覺。
```

## 六、前端設計

### 避免「AI 風格」
Claude 預設會產出通用的設計。要獲得更好的前端：

```
專注於：
- 字體：選擇美觀獨特的字體，避免 Arial、Inter
- 色彩：承諾一致的美學，用 CSS 變數保持一致性
- 動畫：使用 CSS 動畫和微互動
- 背景：創造氛圍和深度，不要預設純色

避免：
- 過度使用的字體（Inter、Roboto、Arial）
- 陳腔濫調的配色（白底紫漸層）
- 可預測的佈局和元件模式
```

## 八、13 語言 Claude Code 效能基準測試（新）

來源：InfoQ「Dynamic Languages Faster and Cheaper in 13-Language Claude Code Benchmark」

重要發現：
- **動態語言（Python、JS）在 Claude Code 中更快且更便宜**
- 編譯語言（Rust、Go）在某些場景下 token 消耗更高
- 選擇正確的語言可以顯著影響 Claude Code 的效率和成本

**可馬上用的建議：**
- 原型開發和腳本任務優先用 Python / JavaScript
- 只在效能關鍵的場景才讓 Claude Code 用編譯語言
- 根據任務類型選擇語言，而非習慣

**難度：** 進階
